High-Dimensional Feature Scaling in PennyLaneAdversarialProcess

Investigation backing NEXT_STEPS.md gap 3 (I=1, E=2, S=1 → 2) — every number and chart below is computed from a real, reproducible sweep of 61 live PennyLane lightning.qubit circuit trainings (no mocked data), run in 278s.
Why this exists: the plan explicitly deferred gap 3 — “only act if >50 features hurt accuracy” — and flagged two candidate fixes without picking one. Before writing code, the trigger condition itself needed checking; after writing code, the first candidate fix (auto-raising num_layers) turned out not to work and was replaced. This report is the evidence trail for both calls.

1. Does circuit position matter, independent of dimensionality?

At input_dim=16 with num_qubits=4, num_layers=4 (weights_elements=48), num_reup=3 — well above the redundancy floor, so neither the pre-fix truncation bug nor the current PCA mitigation ever triggers here. The only thing that differs between the two groups below is which column (0 or 15) carries the discriminative signal.

Mean accuracy: first column 84.0% (seeds [1, 2, 3, 4, 5]: [0.767, 1.0, 1.0, 0.567, 0.867]) vs. last column 75.3% ([0.867, 0.8, 0.933, 0.567, 0.6]). forward() reshapes the (tiled) input vector directly into the (n_layers, n_wires, 3) weight tensor via torch.reshape, so a feature's column index fixes which specific rotation gate it multiplies into. Some positions sit on a more favorable gradient landscape at random init than others — a real effect, and one that no dimensionality-reduction step can undo, because PCA components still have to land somewhere in that same reshape.

2. Does accuracy actually degrade with input_dim, and does the PCA fix help?

Sweeping input_dim from 8 to 100 on the same fixed circuit, comparing the genuine pre-fix code (git show HEAD:pbg_pennylane_adversarial/processes.py, no mitigation) against the current code (PCA mitigation, MIN_NUM_REUP=2). 3 seeds per point ([1, 2, 3]).

The trigger condition holds: at input_dim=8 both variants average 97.8% (well-conditioned, no mitigation needed by either), but by input_dim=48 the pre-fix code averages 45.6% and the PCA-fixed code averages 47.8% — both near chance, and not meaningfully different from each other at this dimension. At input_dim=24 — exactly where the current code's PCA trigger sits — the two variants are identical (57.8% vs. 57.8%) because PCA hasn't kicked in yet (24 is the threshold, not yet exceeded). Looking across the full sweep, the PCA-fixed line sits at or slightly above the pre-fix line at most dimensions, but the gap is small relative to the seed-to-seed spread (visible as the scattered dots behind each mean line) — PCA is not pulling accuracy back up to the input_dim=8 ceiling.

3. The rejected fix: does more training rescue a bigger circuit?

The plan's first suggested fix was auto-raising num_layers to restore num_reup. Manually forcing num_layers=8 at input_dim=48 (which raises weights_elements to 96, restoring num_reup=2) and sweeping the training budget from 8 to 50 epochs, 3 seeds per point ([1, 2, 3]):

Mean accuracy at 8 epochs: 53.3%. At 50 epochs — a 6x larger training budget — mean accuracy: 46.7%. No recovery. This is why the auto-raise-num_layers approach was dropped in favor of PCA: growing the circuit doesn't just cost more compute, it doesn't converge to a better answer even when given far more of that compute, consistent with a barren-plateau-type effect in deeper StronglyEntanglingLayers circuits.

Synthesis

Three independent effects are visible in this data, and only one of them is addressed by the shipped fix:

  • Redundancy collapse (num_reup ≤ 1) — the effect the plan named. PCA addresses this directly by keeping input_dim within the circuit's capacity.
  • Position sensitivity (Section 1) — not addressed by PCA or any fix implemented here. A component that captures the discriminative signal can still land on an unfavorable reshape position.
  • Circuit-depth trainability (Section 3) — the reason growing the circuit was rejected as a fix; PCA avoids this by construction (it never changes num_qubits/num_layers), but doesn't reverse it either.
Honest caveat. Section 2's own data shows the PCA fix's effect size is small relative to seed-to-seed noise at the dimensions where it's needed most (input_dim=48: 45.6% unfixed vs. 47.8% fixed). It guarantees the necessary mathematical condition (num_reup ≥ MIN_NUM_REUP) but not the sufficient one (good accuracy), because Sections 1 and 3's effects are untouched by it. This is harm reduction on a genuinely hard optimization problem, appropriately scoped to a low-priority (I=1, E=2, S=1) chunk — not a claim that high-dimensional datasets will now train well.
