pbg-pennylane-adversarial (this repo) is the stronger vehicle for the WCM use case on all four axes. It has run its quantum classifier against confirmed real v2ecoli data, published an honest negative result (classical baselines match/beat it), and used that result to drive a researched, citation-backed roadmap (USE_CASES.md) with a hard kill-gate before the next multi-session investment (A1 / QGRNN, phase 4). pbg-pennylane-data-reuploading has broader dashboard/scaffolding investment and one genuinely novel component (NonlinearProcessGenerator), but its own same-day self-audit found the core biological claims unvalidated, one process's scientific framing actively misleading, and a barely-above-random real-data result (AUC-ROC 0.625) nowhere acknowledged in its research artifacts.

2The single most load-bearing fact

Both repos ran their quantum classifier against real v2ecoli whole-cell-model data. The results and how each project handled them differ sharply.

4What data-reuploading's own audit says

A deep-breath self-review was run against pbg-pennylane-data-reuploading on the same day as this comparison (.perspective.md, 2026-07-02). It is unusually candid, and its findings independently corroborate the gap identified here — quoted directly rather than paraphrased.

"The code is solid engineering, but the science has a credibility gap... the synthetic data generators are trivial, the quantum circuits are textbook PennyLane, and no process has been validated against a non-trivial benchmark." .perspective.md, Executive summary
"No quantum advantage claim can currently be defended because the quantum and classical baselines don't share data splits, and the real v2ecoli data achieves only AUC-ROC 0.625 (barely above random)." .perspective.md, Executive summary
"'Entanglement = pathway cross-talk' claim is repeated without evidence — the circuits use a generic CNOT chain, not a biologically-informed topology." .perspective.md, §1 Stated purpose

5Where data-reuploading is genuinely ahead

The comparison isn't one-sided in every respect — worth naming plainly.

  • Dashboard/tooling investment. Integrated with vivarium-dashboard (registry browsing, live build_core() introspection, one-click branch merge/PR), which pbg-pennylane-adversarial doesn't have — lowers the barrier for a non-CLI collaborator to poke at composites.
  • Process count / architectural breadth. Five distinct Process classes vs. one — more surface area if any single idea turns out to be worth pursuing, though breadth is also why the science is thin in any one place.
  • NonlinearProcessGenerator. A controlled benchmark with a known (if not yet empirically verified) ground-truth decision boundary is a legitimately useful tool for isolating "does the quantum circuit learn anything" from "is the real data too noisy to learn from" — something pbg-pennylane-adversarial doesn't have an equivalent of, and could arguably borrow.
  • Structured investigation scaffolding. The study.yaml v4 narrative spine (Design → Build → Simulate → Evaluate → Decide phases) is a more formalized research-tracking format than this repo's flatter NEXT_STEPS.md/todo.md pairing.

6Recommendation

Both repos live under the same GitHub org (vivarium-collective) and are visible to the same community, so this is a resource-allocation question, not a build-vs-discard one.

  • Primary WCM quantum-ML effort should stay in pbg-pennylane-adversarial. It has the evidence trail, the honest-negative-result culture, and a concretely scoped next step (A1 phase 4) with a built-in kill-gate — exactly the discipline the other project's own audit says is missing.
  • Salvage, don't duplicate, from data-reuploading: port NonlinearProcessGenerator's controlled-benchmark idea (or the concept of it) into this repo's evaluation toolkit — it directly answers "is there signal to find at all," which is exactly what A1's phase-4 decision gate is trying to establish for a different architecture.
  • Do not extend data-reuploading's biological claims (pathway-informed entanglement, GT2P, minimal-genome) until its own audit's items 1 and 3 are resolved — shared data splits and a documented real-data cross-validation result. Building more Process classes on an unvalidated foundation compounds the credibility gap rather than closing it.
